业务列表
1. 账号与认证业务
这是 App 侧的入口业务，负责监护人账号的生命周期管理。这里不仅仅是注册登录，还要把“谁有权限看哪个老人、哪个设备”这件事的基础打好。
这个业务下需要设计的内容主要包括：注册、登录、找回密码、登录态管理、账号安全策略，以及账号与老人档案、监护关系之间的关联规则。
2. 老人档案与监护关系业务
这个业务负责把“设备服务谁、数据属于谁、推送发给谁”定义清楚。老人档案不是普通资料页，而是整个平台的数据归属中心。
这里要设计老人基础信息、健康相关信息、紧急联系人、一个老人对应多个监护人、主监护人与普通监护人的权限区别等内容。
3. 设备接入与安全注册业务
这是硬件进入系统的第一步，负责解决“这个设备是不是合法设备”的问题。你给的 /device_register 就属于这个业务。
这里要设计设备首次注册、合法性校验、设备基础档案建立、重复注册处理、注册失败处理，以及后续是否允许设备重新激活等规则。
4. 设备在线状态与心跳管理业务
这个业务负责判断设备是否在线、何时离线、当前是否可用。它是实时监控和绑定管理的支撑能力。
这里要设计心跳上报、在线判定、离线判定、设备状态刷新、异常掉线处理、设备最后在线时间、设备运行状态展示等。
5. 设备绑定与解绑业务
这是整个平台最核心的业务之一，因为后续所有数据分发、历史查询、告警推送都依赖绑定关系。
这里要设计在线设备搜索、绑定发起、绑定确认、唯一绑定约束、解绑流程、换绑流程、绑定冲突处理、设备未绑定时的行为限制等。
6. 设备数据接收与存储业务
这个业务负责承接硬件上传的数据，并把不同类型的数据正确归档。它是“平台数据底座”。
这里要设计硬件数据分类、上传方式、数据落库策略、图片和音频文件存储、定位数据存储、告警数据存储、历史查询能力，以及数据和绑定关系的关联方式。
7. 告警事件管理业务
这是产品价值最直接的一层，尤其是摔倒和避障危险告警。这个业务的重点不只是“收到了告警”，而是“能不能及时、准确、可追溯地通知监护人”。
这里要设计告警类型、告警触发条件、告警去重机制、告警等级、告警入库、告警状态、历史记录、告警详情展示、告警处理闭环等。
8. 定位与设备状态展示业务
这个业务面向 App 展示层，让监护人看到老人当前所在位置和设备运行情况。它和告警业务有关联，但又应当独立设计。
这里要设计最新位置展示、历史轨迹展示、设备在线状态展示、设备基础运行数据展示，以及定位与告警之间的关联查看方式。
9. 药品图片识别与智能建议业务
这是 AI 相关的核心亮点业务，业务链路比普通上传复杂，包含图片、OCR、语义理解和建议生成多个阶段。
这里要设计图片上传、OCR 任务处理、识别文本存储、识别失败处理、药品信息展示、LLM 建议生成、结果审核与风险提示、历史识别记录等。
10. 消息推送与通知业务
这个业务负责把平台事件真正送达到监护人。没有这一层，前面的告警和识别结果就无法形成有效触达。
这里要设计推送触发规则、推送对象选择、App 弹窗、铃声提醒、消息中心、未读已读状态、推送失败补偿，以及不同类型消息的优先级策略。

## 一、认证服务

### 1.设备验证服务
每台设备出厂时应至少内置以下信息：

- device_id：设备唯一编号，业务主标识
- device_code：设备密钥材料，等价于设备密码，不应该暴露给用户

#### i.设备身份认证
设备首次启动、JWT 失效、或重连时，请求认证挑战接口。
业务逻辑： 设备提供设备码 -> 服务端返回时间戳和过期时间 -> 使用加密函数加密之后建立一个record，challengeId作为key，计算好的加密值作为value -> 把record存入redis -> 设备使用加密函数加密之后返回加密值 -> 服务端查redis看challengeId是否过期，若过期返回错误码 ->  看查出的值与请求传入的值是否一致，若不一致返回验证错误 -> 返回校验成功jwt

record:
key: "challengeId"
value: "计算好的加密值"

请求体:
```JSON
{
    "deviceId": "DEVICE_001" //设备码
}
```
服务端返回：
```JSON
{
  "challengeId": "c_8f3a2d", //此次业务编号
  "nonce": "a8KpQ21xYz", //随机因子
  "timestamp": 1777292501 //现在时间戳
}
```
请求体:
```JSON
{
    "deviceId": "DEVICE_001",  
    "challengeId": "c_8f3a2d",
    "sigin":"计算好的加密值"
}
```
服务端返回：
```JSON
    "jwt":"具体校验成功的jwt"
```
#### ii.设备首次接入注册
设备首次接入注册用于把设备初始化登记到云端系统中。该流程发生在设备第一次联网或首次接入当前平台时，用于确认设备是否合法、是否已在平台预置，以及记录设备基础信息。
业务逻辑：
设备发起首次接入注册请求 -> 服务端根据 deviceId 查询设备档案 -> 若设备不存在，返回设备未登记错误 -> 若设备存在但已被禁用，返回设备不可用错误 -> 若设备合法，记录首次接入时间、最近接入时间、设备型号、固件版本、网络信息等 -> 若该设备已注册，则本次不重复创建设备档案，只更新最近接入信息 -> 返回注册成功结果 -> 设备进入待认证状态
record:
key: "deviceId"
value: "设备基础信息"

请求体:
```json
{  "deviceId": "DEVICE_001",  "deviceModel": "ESP32_K210",  "firmwareVersion": "1.0.0",  "ip": "192.168.1.10"}
```

服务端返回：
```json
{  "code": 0,  "message": "register success"}
```
#### iii.设备 JWT 签发与校验。
设备完成身份认证后，服务端向设备签发 JWT。设备后续访问受保护接口时，需在请求头中携带该 JWT。服务端负责对 JWT 的合法性、有效期和设备身份进行校验。
业务逻辑：
设备通过身份认证 -> 服务端生成 JWT -> 将 deviceId、签发时间、过期时间等信息写入 JWT -> 返回 JWT 给设备 -> 设备后续访问业务接口时携带 JWT -> 服务端校验 JWT 签名、有效期、设备身份 -> 若校验通过则允许访问 -> 若 JWT 失效则返回错误码，要求设备重新执行身份认证
```JSON
{  "deviceId": "DEVICE_001"}
```
```JSON
{
  "jwt": "具体校验成功的jwt"
}
```
#### iV.设备在线状态维护。
设备在线状态维护用于实时反映当前设备是否处于在线可用状态，为绑定、告警、定位和监控业务提供支撑。
业务逻辑：
设备认证成功并发送首次有效心跳 -> 服务端将设备状态标记为在线 -> 后续每次心跳到达时刷新在线状态 -> Redis 中维护实时在线状态 -> PostgreSQL 中维护最近在线时间和最终状态 -> 若长时间未收到心跳，则将设备状态置为离线
#### V.设备心跳处理。
设备心跳处理用于维持设备在线状态，并同步设备轻量运行信息，例如电量、信号强度、当前位置等。
业务逻辑：
设备按固定时间周期发送心跳请求 -> 服务端校验 JWT -> 校验通过后刷新 Redis 中的在线状态和最后活跃时间 -> 更新 PostgreSQL 中的最后心跳时间、最近在线时间、电量、信号强度等 -> 若设备原状态为离线，则恢复为在线并记录一次重连事件 -> 返回心跳接收成功结果
```JSON
{  "deviceId": "DEVICE_001",  "timestamp": 1777292550,  "battery": 78,  "wifiRssi": -56,  "longitude": 115.123456,  "latitude": 28.123456}
```
```JSON
{  "code": 0,  "message": "heartbeat success"}
```
#### Vi.设备断线判定。
设备断线判定用于判断设备是否因网络异常、掉电或其他原因失去连接。
业务逻辑：
服务端定时任务扫描设备最近心跳时间 -> 若超过设定时间未收到设备新心跳 -> 判定设备离线 -> 更新 Redis 中设备在线状态为 offline -> 更新 PostgreSQL 中最近离线时间和运行状态 -> 记录断线日志 -> 必要时投递离线事件供通知服务处理
#### Vii.设备重连处理。
设备重连处理用于支持设备在断网恢复、重启或 JWT 失效后重新恢复接入能力。
业务逻辑：
设备恢复网络后检查本地 JWT 是否有效 -> 若 JWT 有效，则直接恢复发送心跳 -> 若 JWT 已过期，则重新请求身份认证 -> 服务端在收到离线设备的首个有效请求后将其状态恢复为在线 -> 记录重连时间和重连日志 -> 若设备频繁重连，则记入异常日志
#### Viii.设备基础信息记录。
设备基础信息记录用于维护设备静态资料和最近运行状态，为设备管理、绑定关系和问题排查提供支持。
业务逻辑：
设备首次注册时写入基础信息 -> 后续在认证、心跳、重连等场景中更新动态信息 -> 服务端统一维护设备主档案 -> 提供给其他业务模块使用，例如绑定服务、告警服务、设备搜索服务
```JSON
{
  "deviceId": "DEVICE_001",
  "deviceModel": "ESP32_K210",
  "firmwareVersion": "1.0.0"
}
```
```JSON
{
  "code": 0,
  "message": "record success"
}
```
#### iX.设备认证日志与异常日志
设备认证日志与异常日志用于记录设备接入过程中的正常行为和异常行为，便于后续审计、排障与安全分析。
业务逻辑：
设备每次发起认证、注册、心跳、重连、离线状态变更时，服务端记录对应日志 -> 对 challenge 过期、签名不一致、JWT 失效、频繁重连、非法设备访问等异常行为单独记录异常日志 -> 日志写入 PostgreSQL 或日志系统 -> 提供后续查询和排查能力
```JSON
{
  "deviceId": "DEVICE_001",
  "logType": "auth_fail",
  "message": "signature mismatch"
}
```
```JSON
{
  "code": 0,
  "message": "log success"
}
```
### 2. 客户端认证模块
客户端认证模块面向平台用户（如管理员、终端消费者），提供账号注册、登录、会话维持与权限校验等能力，是实现用户操作设备、查看数据的前提。

#### i. 用户注册
业务逻辑：
用户提交注册信息 → 服务端校验用户名、邮箱、手机号是否已存在 → 校验密码复杂度 → 对密码加盐哈希后落库 → 用户状态置为待激活或正常（视是否需要邮箱/短信验证而定） → 返回注册成功结果。

请求体：
```json
{
  "username": "user_001",
  "password": "P@ssw0rd!",
  "email": "user@example.com",
  "phone": "13800138000"
}
```
服务端返回：

```json
{
  "code": 0,
  "message": "register success"
}
```
#### ii. 用户登录认证
业务逻辑：
用户提交登录凭证（用户名+密码或短信验证码） → 服务端根据用户名查找用户记录 → 检查账户是否锁定/禁用 → 比对密码哈希 → 若验证通过，签发 access_token（JWT）和 refresh_token → 将 refresh_token 哈希后存入 Redis，记录关联用户与过期时间 → 返回令牌。

请求体（密码登录）：

```json
{
  "username": "user_001",
  "password": "P@ssw0rd!"
}
```
服务端返回：

```json
{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "dGhpcyBpcyBh...",
  "expires_in": 3600
}
```
#### iii. Token 刷新
业务逻辑：
客户端在 access_token 过期前使用 refresh_token 获取新令牌 → 服务端校验 refresh_token 是否在 Redis 中且未过期、是否属于同一用户 → 若校验通过，使当前 refresh_token 失效（防重放）并签发新的 access_token 和 refresh_token（轮换机制） → 返回新令牌。

请求体：

```json
{
  "refresh_token": "dGhpcyBpcyBh..."
}
```
服务端返回：

```json
{
  "access_token": "eyJhbGciOi...",
  "refresh_token": "bmV3IHJlZnJl...",
  "expires_in": 3600
}
```
#### iv. 用户权限校验
业务逻辑：
用户访问受保护接口时在 Headers 中携带 Authorization: Bearer {access_token} → 网关或服务端拦截器校验 JWT 签名、有效期、签发者 → 从 JWT 中提取用户 ID 与角色 → 比对该接口所需权限 → 校验通过则放行，否则返回 403 无权限。

校验失败返回：

```json
{
  "code": 403,
  "message": "insufficient permission"
}
```
### v. 用户登出
业务逻辑：
客户端提交当前 refresh_token → 服务端将 refresh_token 标记为失效（从 Redis 中删除） → 对应 access_token 仍会自然过期，如需即时失效，可将 JWT jti 加入黑名单缓存（短期保留，至 JWT 原始过期时间）。

请求体：

```json
{
  "refresh_token": "bmV3IHJlZnJl..."
}
```
服务端返回：

```json
{
  "code": 0,
  "message": "logout success"
}
```
#### vi. 账户安全策略
登录保护：连续失败 8 次后锁定账户或要求验证码，记录异常登录日志。

多因素认证（MFA）：可扩展短信/邮件/TOTP 验证。

异地/新设备登录识别：结合 IP、User-Agent 生成风险提示或二次验证。

## 二、老人档案与监护关系业务
本模块以监护人账号为核心，每个注册用户即是一个监护人主体，可在自己账号下创建并管理多个相互独立的老人档案。每个老人档案内可邀请其他用户作为协作监护人，并区分创建者（主监护人）与协作者（普通监护人）的权限。设备、数据与推送皆按“账号 → 老人档案”的归属链路流动。

### 1. 创建老人档案
业务逻辑：
已登录用户（监护人）提交老人信息 → 服务端校验用户账号状态 → 为该用户创建一条老人档案记录 → 自动将创建者设为主监护人（角色：primary） → 初始化健康信息子表 → 返回档案 ID。新档案与同一账号下其他档案完全独立。

请求体：

```json
{
  "name": "张奶奶",
  "gender": "female",
  "birthDate": "1945-03-12",
  "idCard": "310***********1234",
  "bloodType": "A",
  "allergy": "青霉素",
  "medicalHistory": "高血压、糖尿病",
  "emergencyContacts": [
    {
      "name": "张小明",
      "relation": "儿子",
      "phone": "139****0000"
    }
  ]
}
```
服务端返回：

```json
{
  "code": 0,
  "message": "create success",
  "data": { "elderId": "ELDER_001" }
}
```
### 2. 查询老人档案
业务逻辑：
请求者必须是该老人档案的监护人（主/普通） → 根据 elderId 查询并返回老人基础信息、健康子表、紧急联系人列表、当前监护人列表（含角色） → 主监护人可查看完整监护人详情，普通监护人仅可看到主监护人姓名。

请求体：

```json
{
  "elderId": "ELDER_001"
}
```
服务端返回：

```json
{
  "code": 0,
  "data": {
    "elderId": "ELDER_001",
    "name": "张奶奶",
    "gender": "female",
    "birthDate": "1945-03-12",
    "bloodType": "A",
    "allergy": "青霉素",
    "medicalHistory": "高血压、糖尿病",
    "emergencyContacts": [
      { "name": "张小明", "relation": "儿子", "phone": "139****0000" }
    ],
    "guardians": [
      { "userId": "U001", "nickname": "小明", "role": "primary" },
      { "userId": "U002", "nickname": "小红", "role": "normal" }
    ],
    "createdBy": "U001",
    "createdAt": "2025-01-15T08:00:00Z"
  }
}
```
### 3. 更新老人基础与健康信息
业务逻辑：
仅该档案的主监护人有权更新 → 校验用户与档案的关系 → 更新姓名、住址、血型、过敏史、病史等允许修改字段 → 记录变更日志，普通监护人不可修改，可查看。

请求体：

```json
{
  "elderId": "ELDER_001",
  "bloodType": "B",
  "allergy": "头孢类,青霉素",
  "address": "幸福小区 5-101"
}
```
服务端返回：

```json
{
  "code": 0,
  "message": "update success"
}
```
### 4. 邀请协作监护人
业务逻辑：
主监护人发起 → 输入被邀请人的手机号/邮箱 → 系统查找该用户是否存在 → 若存在，发送应用内通知及短信提醒；若不存在，生成带有时效的邀请链接（可短信发送），引导对方注册后自动关联 → 被邀请人同意后，关系建立，角色为 normal → 一个老人档案可有多名普通监护人。

请求体：

```json
{
  "elderId": "ELDER_001",
  "invitee": "139****0001",
  "message": "请帮忙照看妈妈"
}
```
服务端返回：

```json
{
  "code": 0,
  "message": "invitation sent",
  "data": { "inviteId": "INV_1001", "expiresAt": "2026-05-03T12:00:00Z" }
}
```
被邀请人接受后，回调或轮询更新状态：

```json
{
  "code": 0,
  "message": "guardian joined",
  "data": { "userId": "U003", "role": "normal" }
}
```

## 三、设备接入与安全注册业务

三、设备接入与安全注册业务
本模块以设备为核心，每一台硬件设备在投入使用前必须先完成安全注册，获得唯一的设备标识（deviceId）及动态密钥。设备注册后处于“待绑定”状态，等待监护人通过老人档案完成绑定。设备上报的所有数据需经过身份认证与传输加密。设备的状态（在线/离线）、固件版本、最后心跳时间等信息由服务端统一管理，并作为后续告警、数据归属的基础。

1. 设备注册（首次激活）
业务逻辑：
设备端在首次上电时，主动调用注册接口。需提供设备序列号（serialNo）、设备型号（model）、MAC地址、硬件版本、固件版本等信息。服务端校验该序列号是否已在白名单中（若开启白名单模式），并检查是否已被注册。注册成功后生成唯一的 deviceId 和初始密钥 deviceSecret（服务端加密存储，返回给设备时使用临时会话密钥保护），同时生成设备证书（可选）。设备注册后状态置为 registered，尚未绑定任何老人档案。

请求体：

json
{
  "serialNo": "SN_ABC_123456",
  "model": "WATCH_Pro_2025",
  "mac": "AA:BB:CC:DD:EE:FF",
  "hwVersion": "1.0",
  "fwVersion": "2.1.0",
  "timestamp": 1714567890000,
  "sign": "md5(serialNo+timestamp+secretPreShared)"
}
服务端返回：

json
{
  "code": 0,
  "message": "device registered",
  "data": {
    "deviceId": "DEV_001",
    "deviceSecret": "d3f9a2c8e7b1...",
    "certificate": "-----BEGIN CERTIFICATE-----\n...\n-----END CERTIFICATE-----",
    "expiresAt": "2027-04-30T23:59:59Z"
  }
}
2. 设备认证（获取接入 Token）
业务逻辑：
已注册的设备每次上线时需调用认证接口，使用 deviceId 和 deviceSecret（或证书）换取临时访问令牌 accessToken。该令牌有效期通常为 24 小时，设备应定期续期。服务端验证设备状态（是否被禁用、是否已绑定且档案未封存）。认证成功后记录设备当前 IP 和首次认证时间，更新在线状态。

请求体：

json
{
  "deviceId": "DEV_001",
  "authType": "secret",
  "authData": "encrypted_secret_or_cert",
  "fwVersion": "2.1.1"
}
服务端返回：

json
{
  "code": 0,
  "message": "authenticated",
  "data": {
    "accessToken": "eyJhbGciOiJIUzI1NiIs...",
    "expiresIn": 86400,
    "serverTime": 1714567890000,
    "bindStatus": "bound",
    "elderId": "ELDER_001"
  }
}
3. 设备心跳上报
业务逻辑：
已认证的设备每隔固定时间（如 30 秒）发送心跳，携带当前电量、网络信号强度、定位信息（可选）、固件版本等。服务端更新设备的 lastHeartbeatTime，并根据心跳丢失情况（如连续 3 次未收到）标记设备离线。若设备绑定的老人档案处于活跃状态，心跳状态会同步更新到监护人仪表盘。

请求体：

json
{
  "deviceId": "DEV_001",
  "accessToken": "eyJhbGciOi...",
  "battery": 87,
  "rssi": -65,
  "location": {
    "lat": 31.2304,
    "lng": 121.4737
  },
  "timestamp": 1714567895000
}
服务端返回：

json
{
  "code": 0,
  "message": "heartbeat ack",
  "data": {
    "serverTime": 1714567895000,
    "nextInterval": 30,
    "configChange": false
  }
}
4. 设备信息更新（由监护人发起）
业务逻辑：
主监护人（或经授权的普通监护人）可修改设备的别名、安装位置、备注等信息，方便识别同一档案下的多台设备。同时支持远程触发设备同步配置（如告警阈值、采样频率）。修改后服务端通过长连接或下次心跳时下发配置更新指令。

## 四、设备在线状态与心跳管理业务

本模块负责实时监控和管理设备的在线/离线状态，是整个平台实时感知能力的基础。设备通过周期性心跳向服务端报告存活状态，服务端根据心跳超时策略判定设备是否离线。在线状态数据同时为绑定管理、告警推送、定位展示等业务提供实时支撑。本模块独立于设备认证模块，专注于"设备是否可用"这一核心问题，确保监护人能够准确了解设备当前运行状况。

### 1. 心跳上报
业务逻辑：
已认证设备按配置的时间间隔（默认 30 秒）向服务端发送心跳请求，携带设备 ID、当前时间戳、电量、信号强度、定位坐标等轻量运行信息 → 服务端校验 JWT 有效性 → 校验通过后更新 Redis 中该设备的在线状态和最后活跃时间戳 → 同步更新 PostgreSQL 中的设备运行快照（电量、信号、位置等） → 若设备当前状态为离线，则恢复为在线并记录一次重连事件 → 返回心跳确认结果及服务端时间。

请求体：
```json
{
  "deviceId": "DEV_001",
  "timestamp": 1714567895000,
  "battery": 87,
  "rssi": -65,
  "location": {
    "lat": 31.2304,
    "lng": 121.4737
  }
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "heartbeat ack",
  "data": {
    "serverTime": 1714567895000,
    "nextInterval": 30,
    "statusChanged": false
  }
}
```

### 2. 在线状态判定
业务逻辑：
服务端维护每台设备的在线状态机，状态包括 online、offline、unknown 三种 → 设备首次有效心跳到达时将状态从 unknown 或 offline 切换为 online → 后续每次心跳刷新在线时间戳 → Redis 中以 key=device:status:{deviceId} 存储实时状态，TTL 设为心跳超时时间的 2 倍 → PostgreSQL 中持久化记录最近一次状态变更时间和当前状态 → 状态变更时发布事件消息供其他业务模块订阅。

请求体：
```json
{
  "deviceId": "DEV_001"
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "deviceId": "DEV_001",
    "status": "online",
    "lastHeartbeat": "2026-05-02T10:30:00Z",
    "continuousOnlineSeconds": 86400
  }
}
```

### 3. 离线判定与超时检测
业务逻辑：
服务端定时任务（每 10 秒执行一次）扫描所有标记为 online 的设备 → 对比当前时间与设备最后心跳时间 → 若超过预设阈值（默认 90 秒，即连续 3 次心跳丢失）仍未收到心跳 → 将设备状态切换为 offline → 更新 Redis 和 PostgreSQL 中的状态 → 记录离线事件日志（包含最后已知位置、电量、信号强度等上下文信息） → 投递离线事件到消息队列 → 通知服务消费离线事件后向该设备关联的老人档案下所有监护人推送离线通知。

请求体（管理端手动查询）：
```json
{
  "deviceId": "DEV_001"
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "deviceId": "DEV_001",
    "status": "offline",
    "lastHeartbeat": "2026-05-02T10:28:30Z",
    "offlineDetectedAt": "2026-05-02T10:30:00Z",
    "lastKnownLocation": {
      "lat": 31.2304,
      "lng": 121.4737
    },
    "lastBattery": 87
  }
}
```

### 4. 异常掉线处理
业务逻辑：
当设备在短时间内频繁出现"上线→离线→上线"的抖动时（例如 5 分钟内超过 3 次状态切换），服务端标记该设备为"连接不稳定" → 记录异常日志，包含抖动时间窗口、切换次数、每次在线/离线时间点 → 向监护人推送网络不稳定预警 → 若异常掉线持续超过 30 分钟，升级为设备故障预警 → 设备恢复稳定在线后，自动清除"连接不稳定"标记并记录恢复时间。

请求体（查询设备连接稳定性）：
```json
{
  "deviceId": "DEV_001",
  "timeRange": {
    "start": "2026-05-02T00:00:00Z",
    "end": "2026-05-02T23:59:59Z"
  }
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "deviceId": "DEV_001",
    "stabilityStatus": "unstable",
    "totalOnlineSeconds": 72000,
    "totalOfflineSeconds": 14400,
    "statusSwitchCount": 5,
    "abnormalFlag": true,
    "recentSwitches": [
      { "from": "online", "to": "offline", "at": "2026-05-02T08:15:00Z" },
```

### 5. 设备最后在线时间查询
业务逻辑：
监护人或系统模块查询设备最后一次成功通信的时间 → 服务端从 Redis 中读取实时在线状态和最后心跳时间 → 若 Redis 中无数据（已过期），则从 PostgreSQL 中读取持久化记录 → 返回最后在线时间、持续在线时长、当前状态等信息 → 该接口为高频查询接口，结果优先从缓存获取。

## 五、设备绑定与解绑业务

本模块是整个平台数据归属和权限流转的核心枢纽。设备绑定建立了"设备→老人档案→监护人"的数据归属链路，后续所有数据上报、告警推送、历史查询都依赖这一绑定关系。绑定操作需确保唯一性约束，即一台设备同一时间只能绑定到一个老人档案。解绑操作需安全可控，解绑后设备历史数据仍保留在原档案中用于健康追溯，设备本身恢复为待绑定状态，可被重新绑定到其他档案。

### 1. 在线设备搜索
业务逻辑：
监护人通过 App 发起设备绑定流程时，需先搜索附近可绑定的在线设备 → 设备端开启蓝牙/WiFi 广播，携带 deviceId 和设备型号 → App 扫描附近设备并展示列表 → 监护人选择目标设备后，App 向服务端查询该设备的注册状态和绑定状态 → 服务端返回设备是否已完成注册、是否已被绑定、设备型号等基本信息 → 仅状态为 registered 且未绑定的设备才允许发起绑定。

请求体：
```json
{
  "deviceId": "DEV_001"
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "deviceId": "DEV_001",
    "model": "WATCH_Pro_2025",
    "registerStatus": "registered",
    "bindStatus": "unbound",
    "firmwareVersion": "2.1.1",
    "canBind": true
  }
}
```

### 2. 绑定发起
业务逻辑：
主监护人（或经授权的普通监护人）选择目标设备和目标老人档案 → 提交绑定请求 → 服务端校验操作者是否为该档案的主监护人或被授权的普通监护人 → 校验设备是否已完成注册且状态为 unbound → 校验老人档案是否处于活跃状态（非 archived） → 校验通过后创建绑定关系记录，状态置为 binding → 向设备端推送绑定通知 → 等待设备确认。

请求体：
```json
{
  "elderId": "ELDER_001",
  "deviceId": "DEV_001",
  "operatorId": "U001"
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "binding initiated",
  "data": {
    "bindId": "BIND_001",
    "status": "pending_device_confirm",
    "expiresAt": "2026-05-02T11:00:00Z"
  }
}
```

### 3. 绑定确认（设备端）
业务逻辑：
设备收到绑定通知后，向服务端发送绑定确认请求 → 服务端校验绑定记录是否存在且未过期 → 校验请求的 deviceId 与绑定记录中的 deviceId 一致 → 将绑定状态更新为 bound → 记录绑定生效时间 → 更新设备的绑定归属（elderId） → 通知监护人 App 绑定成功 → 后续该设备上报的所有数据自动归入该老人档案。

请求体（设备端）：
```json
{
  "deviceId": "DEV_001",
  "bindId": "BIND_001",
  "confirm": true
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "device bind confirmed",
  "data": {
    "bindId": "BIND_001",
    "status": "bound",
    "elderId": "ELDER_001",
    "boundAt": "2026-05-02T10:35:00Z"
  }
}
```

### 4. 唯一绑定约束校验
业务逻辑：
服务端在处理绑定请求时，强制执行唯一绑定约束 → 查询目标设备当前是否已绑定到其他老人档案 → 若已绑定，返回冲突错误，提示当前绑定的档案信息 → 查询目标老人档案是否已有同类型设备绑定（如一个档案仅允许绑定一台主设备） → 若超出限制，返回错误提示 → 所有校验在事务中执行，确保并发安全。

请求体（校验接口）：
```json
{
  "deviceId": "DEV_001",
  "elderId": "ELDER_001"
}
```
服务端返回（冲突场景）：
```json
{
  "code": 4001,
  "message": "device already bound",
  "data": {
    "deviceId": "DEV_001",
    "currentElderId": "ELDER_002",
    "currentElderName": "李爷爷",
    "boundAt": "2026-04-15T09:00:00Z",
    "suggestion": "请先解绑当前设备，再重新绑定"
  }
}
```

### 5. 解绑流程
业务逻辑：
主监护人发起解绑请求 → 服务端校验操作者权限（仅主监护人可操作） → 需二次确认（短信验证码或密码验证） → 校验通过后执行解绑 → 将绑定状态更新为 unbound → 设备恢复为待绑定状态 → 设备历史数据保留在原老人档案中（标记为历史数据，不可删除） → 向设备端推送解绑通知，设备清除本地绑定信息 → 通知档案下所有监护人设备已解绑 → 解绑操作记录永久保存。

请求体：
```json
{
  "elderId": "ELDER_001",
  "deviceId": "DEV_001",
  "verifyCode": "654321",
  "reason": "设备更换"
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "device unbound",
  "data": {
    "deviceId": "DEV_001",
    "newStatus": "unbound",
    "unboundAt": "2026-05-02T11:00:00Z",
    "historyKept": true,
    "historyElderId": "ELDER_001"
  }
}
```

### 6. 换绑流程（解绑+重新绑定）
业务逻辑：
当监护人需要将设备从当前老人档案转移到另一个老人档案时，执行换绑操作 → 先对原档案执行解绑流程（含二次确认） → 解绑成功后，设备状态变为 unbound → 立即对新档案发起绑定请求 → 绑定成功后，设备数据归属切换到新档案 → 原档案中的历史数据保留，新档案从绑定时刻开始接收数据 → 全程记录操作日志，支持审计追溯。

请求体：
```json
{
  "fromElderId": "ELDER_001",
  "toElderId": "ELDER_002",
  "deviceId": "DEV_001",
  "verifyCode": "654321"
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "device rebound success",
  "data": {
    "deviceId": "DEV_001",
    "fromElderId": "ELDER_001",
    "toElderId": "ELDER_002",
    "unboundAt": "2026-05-02T11:00:00Z",
    "boundAt": "2026-05-02T11:00:05Z"
  }
}
```

### 7. 绑定冲突处理
业务逻辑：
当两个监护人同时对同一台设备发起绑定时，服务端采用"先到先得"策略 → 第一个到达的绑定请求正常处理 → 第二个请求返回冲突错误，提示设备已被绑定 → 若绑定处于 pending_device_confirm 状态超时（默认 5 分钟），自动取消该绑定记录，释放设备 → 超时取消后其他监护人可重新发起绑定 → 冲突事件记录日志，供后续分析。

请求体（冲突场景返回）：
```json
{
  "code": 4002,
  "message": "bind conflict, device is being bound by another request",
  "data": {
    "deviceId": "DEV_001",
    "currentBindId": "BIND_002",
    "currentOperator": "U002",
    "suggestion": "请稍后重试或联系当前操作者"
  }
}
```

### 8. 设备未绑定时的行为限制
业务逻辑：
设备处于 unbound 状态时，服务端对设备行为施加限制 → 设备仍可进行心跳上报和认证，维持基本在线状态 → 设备上报的健康数据、定位数据等业务数据会被接收但标记为"未归属"，不触发告警推送 → 设备不可接收远程配置下发 → 设备不可执行固件升级 → 监护人搜索该设备时显示"待绑定"状态 → 设备绑定成功后，未归属期间的数据可选择性地关联到新绑定的老人档案。

请求体（设备上报数据，未绑定状态）：
```json
{
  "deviceId": "DEV_001",
  "timestamp": 1714567900000,
  "type": "heart_rate",
  "value": 98,
  "unit": "bpm"
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "data accepted but unbound",
  "data": {
    "dataId": "data_789012",
    "bound": false,
    "warning": "device is not bound to any elder profile, data will not trigger alerts"
  }
}
```

### 9. 查询设备绑定关系
业务逻辑：
监护人或系统模块查询某台设备当前的绑定关系 → 服务端返回设备当前绑定的老人档案信息、绑定时间、操作者等 → 若设备未绑定，返回 unbound 状态 → 同时返回该设备的历史绑定记录（含已解绑的档案），便于追溯。

请求体：
```json
{
  "deviceId": "DEV_001"
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "deviceId": "DEV_001",
    "currentBinding": {
      "elderId": "ELDER_001",
      "elderName": "张奶奶",
      "boundAt": "2026-04-29T15:30:00Z",
      "boundBy": "U001"
    },
    "bindHistory": [
      {
        "elderId": "ELDER_000",
        "elderName": "测试老人",
        "boundAt": "2026-04-01T10:00:00Z",
        "unboundAt": "2026-04-28T18:00:00Z",
        "reason": "测试结束"
      }
    ]
  }
}
```

## 六、设备数据接收与存储业务

本模块是平台的数据底座，负责接收设备上报的各类数据并正确归档存储。设备数据类型多样，包括结构化的健康指标（心率、血压、步数等）、半结构化的定位数据、非结构化的图片和音频文件等。不同类型的数据采用不同的存储策略：时序数据写入时序数据库，文件类数据存储到对象存储（OSS），关系型数据写入 PostgreSQL。所有数据均与设备绑定关系关联，确保数据归属清晰、查询高效。

### 1. 硬件数据分类与接收
业务逻辑：
设备上报数据时需携带数据类型标识（type 字段），服务端根据类型进行分类处理 → 结构化健康数据（heart_rate、blood_pressure、steps、spo2 等）写入时序数据库 → 定位数据（location）写入时序数据库并同步更新 Redis 中的最新位置缓存 → 告警事件数据（fall、obstacle、sos 等）写入 PostgreSQL 并触发告警业务流程 → 图片/音频数据先上传至 OSS 获取文件 URL，再将 URL 和元数据写入 PostgreSQL → 所有数据均关联 deviceId 和 elderId（若已绑定）。

请求体（健康数据示例）：
```json
{
  "deviceId": "DEV_001",
  "timestamp": 1714567900000,
  "type": "heart_rate",
  "value": 98,
  "unit": "bpm",
  "metadata": {
    "confidence": 0.98,
    "source": "optical"
  }
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "data accepted",
  "data": {
    "dataId": "data_123456",
    "type": "heart_rate",
    "storedAt": 1714567901000,
    "elderId": "ELDER_001"
  }
}
```

### 2. 图片文件上传与存储
业务逻辑：
设备采集图片（如药品照片、环境照片）后，先通过文件上传接口将图片二进制数据上传至 OSS → OSS 返回文件访问 URL → 设备再将文件 URL 和图片元数据（拍摄时间、图片类型、关联业务等）通过数据上报接口发送给服务端 → 服务端将元数据和 URL 写入 PostgreSQL 的文件记录表 → 图片按日期分目录存储，路径规则为 oss://bucket/{elderId}/{year}/{month}/{day}/{fileId}.jpg → 支持图片压缩和缩略图生成。

请求体（文件上传后上报元数据）：
```json
{
  "deviceId": "DEV_001",
  "timestamp": 1714567900000,
  "type": "image",
  "fileUrl": "https://oss.example.com/ELDER_001/2026/05/02/img_abc123.jpg",
  "thumbnailUrl": "https://oss.example.com/ELDER_001/2026/05/02/img_abc123_thumb.jpg",
  "fileSize": 2048576,
  "imageCategory": "medicine",
  "metadata": {
    "width": 3024,
    "height": 4032,
    "format": "jpeg"
  }
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "image metadata saved",
  "data": {
    "fileId": "FILE_001",
    "fileUrl": "https://oss.example.com/ELDER_001/2026/05/02/img_abc123.jpg",
    "storedAt": 1714567901000
  }
}
```

### 3. 音频文件上传与存储
业务逻辑：
设备采集音频（如语音备忘、环境录音）后，上传流程与图片类似 → 音频文件上传至 OSS → 返回 URL 后上报元数据 → 服务端记录音频文件信息，包括时长、采样率、编码格式等 → 音频文件按业务类型分目录存储 → 对于语音类音频，可触发后续语音识别（ASR）流程，将识别文本关联存储。

## 七、告警事件管理业务

本模块是产品安全价值的核心承载层，负责接收、处理、分发和追踪设备产生的各类告警事件。告警事件直接关系到老人的生命安全，因此本模块的设计重点在于"及时性、准确性、可追溯性"。系统需确保告警从产生到通知监护人的端到端延迟尽可能低，同时通过去重和分级机制避免告警疲劳。每一条告警都形成完整的处理闭环，从触发到确认再到解决，全程可追溯。

### 1. 告警类型定义
业务逻辑：
系统预定义以下告警类型，每种类型对应不同的触发条件和处理策略 → 摔倒告警（fall）：设备加速度传感器检测到疑似摔倒动作，需结合姿态角和冲击力综合判断 → 避障危险告警（obstacle）：设备超声波/红外传感器检测到前方有障碍物且距离低于安全阈值 → SOS 紧急呼叫（sos）：老人手动按下 SOS 按钮触发 → 心率异常告警（heart_rate_abnormal）：心率持续超出预设安全范围 → 低电量告警（low_battery）：设备电量低于 20% → 离线告警（device_offline）：设备超过预设时间未上报心跳 → 电子围栏告警（geofence）：设备定位超出预设安全区域。

请求体（查询告警类型列表）：
```json
{}
```
服务端返回：

```json
      { "type": "sos", "name": "紧急呼叫", "defaultLevel": "critical", "autoPush": true },
      { "type": "heart_rate_abnormal", "name": "心率异常", "defaultLevel": "warning", "autoPush": true },

    ]
  }
}
```

### 2. 告警触发与接收
业务逻辑：
设备检测到异常事件后，立即上报告警数据 → 服务端接收告警数据后，首先验证设备身份和绑定关系 → 验证通过后，生成告警记录（包含唯一 alertId、告警类型、等级、触发时间、设备信息、关联老人档案、传感器原始数据等） → 告警记录写入 PostgreSQL → 同时将告警事件投递到消息队列 → 告警处理服务消费消息队列中的告警事件，执行后续去重、分级、推送流程 → 从设备上报到生成告警记录的端到端延迟应控制在 500 毫秒以内。

### 3. 告警去重机制
业务逻辑：
为避免同一事件在短时间内重复触发大量告警（如老人摔倒后设备持续检测到异常姿态），服务端实施告警去重 → 去重规则：同一设备、同一告警类型、在设定时间窗口内（默认 60 秒）的多次告警视为同一事件 → 首次告警正常创建并推送 → 时间窗口内的后续告警被标记为重复，仅更新原始告警的最后触发时间，不重复推送 → 去重窗口可根据告警类型配置，摔倒告警窗口为 120 秒，心率异常窗口为 300 秒 → 去重计数记录在原始告警中，便于分析事件持续时间。


服务端返回：

### 4. 告警等级与分级策略
业务逻辑：
告警分为四个等级，每个等级对应不同的推送策略和处理时效要求 → critical（紧急）：摔倒、SOS 等危及生命安全的事件，立即推送所有监护人，支持电话语音通知，5 分钟内未确认则升级推送 → warning（警告）：避障危险、心率异常、设备离线等需关注的事件，推送所有监护人，30 分钟内未确认则二次提醒 → info（提示）：低电量、固件升级等一般性通知，仅推送主监护人，不强制确认 → notice（公告）：系统维护、功能更新等平台级通知，推送所有用户，无需确认 → 监护人可在 App 中自定义各类型告警的推送等级和免打扰时段。

### 8. 告警处理闭环
业务逻辑：

## 八、定位与设备状态展示业务

## 九、药品图片识别与智能建议业务

本模块是平台的 AI 核心亮点业务，通过图像识别和自然语言处理技术为老人提供用药安全辅助。业务链路涵盖图片上传、OCR 文字识别、药品信息匹配、大语言模型（LLM）用药建议生成等多个阶段。整个流程采用异步处理模式，前端提交识别请求后通过轮询或 WebSocket 获取处理结果。识别结果包含药品名称、用法用量、注意事项等结构化信息，以及 LLM 生成的个性化用药建议。系统对识别结果进行风险标注，提醒监护人关注潜在用药风险。

### 1. 图片上传
业务逻辑：
监护人通过 App 拍摄或从相册选择药品图片（药盒、说明书、药片等）→ 图片经客户端压缩后上传至服务端 → 服务端接收图片并存储到 OSS → 生成唯一的图片 ID（imageId）和访问 URL → 同时生成缩略图用于列表展示 → 将图片元数据（imageId、URL、大小、拍摄时间、上传者等）写入 PostgreSQL → 返回 imageId 给客户端，用于后续查询识别结果 → 图片支持 JPEG、PNG 格式，单张大小限制 10MB。

请求体（图片上传后上报元数据）：
```json
{
  "elderId": "ELDER_001",
  "deviceId": "DEV_001",
  "imageCategory": "medicine",
  "fileUrl": "https://oss.example.com/ELDER_001/2026/05/02/med_abc123.jpg",
  "thumbnailUrl": "https://oss.example.com/ELDER_001/2026/05/02/med_abc123_thumb.jpg",
  "fileSize": 2048576,
  "metadata": {
    "width": 3024,
    "height": 4032,
    "format": "jpeg",
    "source": "camera"
  }
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "image uploaded",
  "data": {
    "imageId": "IMG_001",
    "fileUrl": "https://oss.example.com/ELDER_001/2026/05/02/med_abc123.jpg",
    "thumbnailUrl": "https://oss.example.com/ELDER_001/2026/05/02/med_abc123_thumb.jpg",
    "uploadedAt": "2026-05-02T11:00:00Z"
  }
}
```

### 2. OCR 任务处理
业务逻辑：
图片上传成功后，客户端发起 OCR 识别请求 → 服务端创建 OCR 异步任务，生成 taskId → 将任务投递到消息队列 → OCR Worker 消费任务，调用 OCR 引擎对图片进行文字识别 → OCR 引擎返回识别出的文本内容和置信度 → Worker 将识别结果写入 PostgreSQL，关联 imageId → 更新任务状态为 completed → 若 OCR 识别失败（图片模糊、无文字等），任务状态置为 failed，记录失败原因 → 客户端通过 taskId 轮询任务状态获取结果。

请求体（发起 OCR 识别）：
```json
{
  "imageId": "IMG_001",
  "language": "zh"
}
```
服务端返回：
```json
{
  "code": 0,
  "message": "ocr task created",
  "data": {
    "taskId": "OCR_001",
    "status": "processing",
    "estimatedTime": 5
  }
}
```

### 3. 识别文本存储与药品匹配
业务逻辑：
OCR 识别完成后，服务端对识别出的文本进行药品信息匹配 → 从识别文本中提取药品名称、规格、用法用量等关键信息 → 在药品数据库中进行模糊匹配，找到最相似的药品记录 → 返回匹配的药品信息，包括通用名、商品名、成分、适应症、用法用量、禁忌症等 → 若匹配到多个候选药品，按相似度排序返回 → 匹配结果与 imageId、elderId 关联存储 → 识别文本原文也一并存储，供后续审核和优化使用。

```json
}
```

### 4. 识别失败处理
业务逻辑：
当 OCR 识别失败或药品匹配失败时，系统进行容错处理 → OCR 失败原因包括：图片模糊、光线不足、非药品图片、文字过小等 → 服务端记录失败原因并返回给客户端 → 客户端提示用户重新拍摄或手动输入药品名称 → 对于药品匹配失败的情况（OCR 成功但未找到匹配药品），将识别文本存入待审核队列 → 人工审核员或 LLM 辅助审核后补充药品数据库 → 审核完成后通知提交者 → 连续失败超过 3 次的设备标记为异常，提醒监护人检查设备摄像头。

请求体：
```json
{
```

服务端返回：

## 十、消息推送与通知业务

本模块是平台事件触达监护人的最终通道，负责将告警事件、设备状态变更、识别结果等各类平台事件及时、准确地推送到监护人。推送通道包括 App 内通知（弹窗、横幅）、系统级推送（APNs/FCM）、短信和电话语音通知。不同类型的事件采用不同的推送策略，确保紧急事件优先送达。本模块还负责消息的持久化存储、未读/已读状态管理、推送失败补偿等，确保每一条重要消息都不会遗漏。

### 1. 推送触发规则
业务逻辑：
系统定义各类事件的推送触发规则，当事件发生时自动触发推送 → 告警事件：critical 级别立即推送所有监护人 + 电话语音通知；warning 级别推送所有监护人；info 级别仅推送主监护人 → 设备状态变更：设备离线推送所有监护人；设备上线不推送（避免打扰）；设备低电量推送主监护人 → 识别结果：高风险药品识别结果推送所有监护人；低风险结果不主动推送，仅更新消息中心 → 系统通知：平台维护、版本更新等推送所有用户 → 所有推送规则支持监护人自定义配置。

请求体（查询推送规则配置）：
```json
{
  "elderId": "ELDER_001",
  "userId": "U001"
}
```
服务端返回：
```json
{
  "code": 0,
  "data": {
    "pushRules": {
      "alert": {
        "critical": { "push": true, "channels": ["app", "sms", "voice_call"], "allGuardians": true },
        "warning": { "push": true, "channels": ["app", "sms"], "allGuardians": true },
        "info": { "push": true, "channels": ["app"], "allGuardians": false }
      },
      "device": {
        "offline": { "push": true, "channels": ["app", "sms"], "allGuardians": true },
        "online": { "push": false },
        "low_battery": { "push": true, "channels": ["app"], "allGuardians": false }
      },
      "medicine": {
        "high_risk": { "push": true, "channels": ["app", "sms"], "allGuardians": true },
        "low_risk": { "push": false }
      }
    }
  }
}
```

### 2. 推送对象选择
业务逻辑：
推送服务根据事件类型和配置规则确定推送目标 → 获取事件关联的老人档案 → 查询该档案下所有监护人列表 → 根据推送规则筛选目标监护人（allGuardians=true 则推送全部，false 则仅推送主监护人）→ 查询每个目标监护人的推送偏好设置（免打扰时段、推送渠道偏好）→ 排除处于免打扰时段的监护人（critical 级别除外）→ 生成推送任务列表，每个任务包含目标用户、推送渠道、消息内容。

请求体（内部接口，推送服务调用）：

业务逻辑：

### 7. 推送失败补偿机制
业务逻辑：
推送服务对每条推送记录投递结果 → 若推送失败（如设备离线、推送证书过期、短信网关超时等），将失败消息加入重试队列 → 重试策略：首次失败后 30 秒重试，第二次失败后 5 分钟重试，第三次失败后 30 分钟重试，最多重试 3 次 → 重试时根据消息时效性判断是否仍需推送（如 1 小时前的告警可能已处理，不再重试）→ 所有重试记录写入推送日志 → 连续失败超过 3 次的消息标记为 delivery_failed，不再自动重试 → 管理员可在后台查看推送失败报表，手动触发重新推送。

请求体（查询推送状态）：
```json
{
  "messageId": "MSG_001"
}
```
服务端返回：
```json
{
  "code": 0,

    "messageId": "MSG_001",

    "attempts": [
      {
        "attempt": 1,
        "channel": "app",
        "status": "failed",
        "reason": "device_offline",
        "at": "2026-05-02T10:30:01Z"

        "channel": "app",
        "status": "delivered",
        "at": "2026-05-02T10:30:31Z"
```

### 8. 消息优先级策略
业务逻辑：
系统根据消息类型和紧急程度设定优先级，确保高优先级消息优先送达 → 优先级从高到低：P0（紧急）→ P1（重要）→ P2（一般）→ P3（低）→ P0：critical 级别告警、SOS 紧急呼叫，使用所有可用通道推送，包括电话语音 → P1：warning 级别告警、设备离线，使用 App 推送 + 短信 → P2：info 级别告警、药品识别结果、低电量提醒，仅 App 推送 → P3：系统通知、固件升级提示，App 推送，可延迟投递 → 推送通道资源紧张时（如短信网关限流），优先保障 P0 和 P1 消息 → 监护人可在设置中调整各类型消息的优先级。
