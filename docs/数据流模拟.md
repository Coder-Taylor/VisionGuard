# 端到端数据流模拟

> 环境就绪后（`docker compose up -d` + `go run cmd/server/main.go`），按顺序逐条跑。
> 每条 curl 都标注了预期返回，用于验证各接口。

## 前置：生成测试数据

```bash
cd backend
go run gendevid/token.go              # 生成 gendevid/register.json
```

---

## 阶段一：设备开机注册（ESP32 → 云端）

### Step 1: 设备注册（XOR Token 认证）

ESP32 上电连 WiFi 后只执行一次。计算 XOR Token → POST JSON。

```
ESP32 视角：
  uniqueCode = "DEVICE_2026_ESP32_K210"
  每字符 ^ 0x4B → 拼接成 token
  POST /device_register  →  期望返回 "ok"
```

```bash
curl -s -X POST http://localhost:3000/device_register \
  -H "Content-Type: application/json" \
  --data-binary @gendevid/register.json
# 预期: ok
```

**后端做了什么**：
1. 收到 JSON → 提取 deviceId, uniqueCode, token
2. 对 uniqueCode 逐字符 XOR 0x4B，比对 token → 一致 ✓
3. 数据库 devices 表：INSERT/UPDATE → `online_status='online'`, `bind_status='unbound'`, `ip='127.0.0.1'`
4. auth_logs 表：INSERT → `log_type='register_ok'`
5. 返回纯文本 `ok`

**验证**：查数据库
```sql
SELECT device_id, online_status, bind_status, ip FROM devices WHERE device_id='DEVICE_001';
-- 预期: DEVICE_001 | online | unbound | 127.0.0.1
```

---

### Step 2: 设备轮询绑定状态

ESP32 未绑定时每 2 秒轮询一次。

```
ESP32 视角：
  GET /check_bind?deviceId=DEVICE_001  →  当前返回 "unbound"（还没人绑）
```

```bash
curl -s "http://localhost:3000/check_bind?deviceId=DEVICE_001"
# 预期: unbound
```

**后端做了什么**：
1. 从 query 取 deviceId
2. 查 bindings 表：`SELECT COUNT(*) FROM bindings WHERE device_id='DEVICE_001'` → 0
3. 返回 `unbound`

---

## 阶段二：监护人注册 & 登录（APP → 云端）

### Step 3: 监护人注册

```bash
curl -s -X POST http://localhost:3000/api/v1/auth/register \
  -H "Content-Type: application/json" \
  -d '{"userId":"caregiver01","password":"123456","phone":"13800000000"}'
# 预期: {"code":0,"message":"register success"}
```

**后端做了什么**：
1. 检查 userId 是否已存在
2. bcrypt 加密密码 → password_hash
3. users 表 INSERT

---

### Step 4: 监护人登录（拿 JWT）

```bash
TOKEN=$(curl -s -X POST http://localhost:3000/api/v1/auth/login \
  -H "Content-Type: application/json" \
  -d '{"userId":"caregiver01","password":"123456"}' | grep -o '"jwt":"[^"]*"' | cut -d'"' -f4)
echo "JWT: $TOKEN"
# 预期: {"code":0,"jwt":"eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9..."}
```

**后端做了什么**：
1. 查 users 表找到用户
2. bcrypt.CompareHashAndPassword 校验
3. 签发 HS256 JWT → 7 天有效 → payload: `{userId, exp, iat}`

---

## 阶段三：设备绑定（APP + ESP32 协作）

### Step 5: APP 查看可绑定设备列表

```bash
curl -s http://localhost:3000/api/v1/devices \
  -H "Authorization: Bearer $TOKEN"
# 预期: {"code":0,"devices":[{"deviceId":"DEVICE_001","onlineStatus":"online","bindStatus":"unbound",...}]}
```

**后端做了什么**：
1. JWT 中间件校验 token → 提取 userId
2. `SELECT * FROM devices WHERE online_status='online' AND bind_status='unbound'`
3. 返回设备列表

---

### Step 6: APP 发起绑定

```bash
curl -s -X POST http://localhost:3000/api/v1/device/bind \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"DEVICE_001"}'
# 预期: {"code":0,"message":"bind success"}
```

**后端做了什么**（事务）：
1. 查 device 状态 → online + unbound ✓
2. bindings 表 INSERT → `user_id='caregiver01', device_id='DEVICE_001'`
3. devices 表 UPDATE → `bind_status='bound'`

---

### Step 7: ESP32 轮询得到绑定成功

```bash
curl -s "http://localhost:3000/check_bind?deviceId=DEVICE_001"
# 预期: bind_success     ← 之前是 unbound，现在变了！
```

```
ESP32 视角：
  轮询发现 "bind_success" → 进入正常工作模式
  → 开始 30 秒心跳、监听按键/摔倒/障碍物事件
```

## 阶段四：正常工作——摔倒告警（完整链路）

### Step 8: ESP32 检测到摔倒 → 上报云端

MPU6050 倾角超阈值 → ESP32 本地扬声器蜂鸣 → 联网 POST 告警。

```
ESP32 视角：
  POST /upload_alarm
  Body: {"deviceId":"DEVICE_001","alarmType":"fall","angleX":-48.25,"angleY":12.36,"longitude":115.123456,"latitude":28.123456}
  → 期望 "ok"
```

```bash
curl -s -X POST http://localhost:3000/upload_alarm \
  -H "Content-Type: application/json" \
  -d '{"deviceId":"DEVICE_001","alarmType":"fall","angleX":-48.25,"angleY":12.36,"longitude":115.123456,"latitude":28.123456}'
# 预期: ok
```

**后端做了什么**：
1. 解析 JSON → alarmType=fall
2. event_logs 表 INSERT：
   ```
   device_id=DEVICE_001, event_type=fall, event_time=now(),
   longitude=115.123456, latitude=28.123456,
   raw_json={"alarmType":"fall","angleX":-48.25,"angleY":12.36,...}
   ```
3. 查 bindings 表 → 找到 user_id=caregiver01
4. notifications 表 INSERT：
   ```
   user_id=caregiver01, device_id=DEVICE_001, alert_type=fall,
   title="摔倒告警", body="老人可能摔倒了，请及时查看！", is_read=false
   ```

---

### Step 9: APP 拉取通知

```bash
curl -s http://localhost:3000/api/v1/notifications \
  -H "Authorization: Bearer $TOKEN"
# 预期: {"code":0,"has_alert":true,"alert_type":"fall","notifications":[{...title:"摔倒告警"...}]}
```

APP 端效果：收到 `has_alert=true` → 弹窗 "老人可能摔倒了，请及时查看！" + 手机铃声。

---

### Step 10: APP 标记已读

```bash
# 先记下通知 ID（从 step 9 返回的 notifications[0].ID）
curl -s -X PUT http://localhost:3000/api/v1/notifications/1/read \
  -H "Authorization: Bearer $TOKEN"
# 预期: {"code":0,"message":"ok"}
```

## 阶段五：药品识别（图片上传→OCR→语音播报）

### Step 11: ESP32 上传图片（无 deviceId 参数！）

```
ESP32 视角：
  K210 拍照完成 → ESP32 收到 JPEG 数据
  POST /upload_image
  Header: Content-Type: image/jpeg
  Body: <JPEG 二进制裸流，无 deviceId>
  → 期望 "ok"
```

硬件不传 deviceId！后端怎么知道哪个设备？—— Step 1 注册时后端已记录设备 IP，这里通过客户端 IP 反查。

```bash
# 用一张随便的 JPEG 模拟（或用 dd 生成假数据）
dd if=/dev/urandom of=/tmp/test.jpg bs=1024 count=10 2>/dev/null
curl -s -X POST http://localhost:3000/upload_image \
  -H "Content-Type: image/jpeg" \
  --data-binary @/tmp/test.jpg
# 预期: ok
```

**后端做了什么**：
1. `c.IP()` → 127.0.0.1
2. 查 devices 表：`WHERE ip='127.0.0.1' ORDER BY last_online_at DESC` → DEVICE_001
3. 存盘：`uploads/DEVICE_001/images/20260430_223045.jpg`
4. ocr_records 表 INSERT：`device_id=DEVICE_001, image_path=..., ocr_text=""`
5. 如果配置了 OCR_SERVICE_URL，会先调 OCR 服务

---

### Step 12: ESP32 获取 OCR 文字（也无 deviceId！）

```
ESP32 视角：
  上传图片完成后立即 GET /get_ocr_text（不带任何参数）
  → 返回纯文本，逐字用本地语音库播报
```

```bash
curl -s http://localhost:3000/get_ocr_text
# 预期: (空字符串，因为没有真实 OCR 服务)
```

**后端做了什么**：
1. `c.IP()` → 127.0.0.1 → 反查 → DEVICE_001
2. `SELECT * FROM ocr_records WHERE device_id='DEVICE_001' ORDER BY created_at DESC LIMIT 1`
3. 有 llm_advice 返回 llm_advice，否则返回 ocr_text
4. 返回纯文本

## 数据流全景回顾

```
  ESP32 (192.168.1.x)                    云端 (Go Fiber)                       APP (手机)
  ════════════════                       ══════════════                       ══════════
  ① POST /device_register ──────────────→ XOR 校验 ✓
     {deviceId, uniqueCode, token}         记录 IP, online=1                   (还未登录)
  
  ② GET /check_bind?deviceId=xxx ───────→ 查绑定 → unbound                  
     (每2秒轮询)
                                                                               ③ 注册/登录
                                                                               ─────────→ JWT
  
                                                                               ④ GET /devices
                                                                               ← Device列表 ─
  
                                                                               ⑤ POST /bind
                                                                               ─────────→ 创建绑定
  
  ⑥ GET /check_bind ────────────────────→ bind_success!
     进入工作模式
  
  ═══ 正常工作 ═══
  
  ⑦ 摔倒检测 → 本地蜂鸣
     POST /upload_alarm ────────────────→ event_logs INSERT
     {fall, angleX, angleY, lat, lng}     notifications INSERT ──────────────→ ⑧ APP弹窗+铃声
  
  ⑨ 按键A → K210拍照
     POST /upload_image ────────────────→ 存盘 + OCR
     (JPEG流, 无deviceId!)                 IP反查 → 关联设备
  
  ⑩ GET /get_ocr_text ─────────────────→ 返回文字
     (无参数!)                             IP反查 → 查OCR记录
  
  ⑪ POST /upload_audio ────────────────→ 存盘
     (WAV流, 无deviceId!)                 IP反查 → uploads/{id}/audio/
```

---

## 注意

- **XOR Token 含不可打印字节**：curl 测试用 `--data-binary @文件`。Go 的 `json.Marshal` 已经正确转义为 `` 等，服务端 `json.Unmarshal` 会还原，校验不受影响。
- **IP 反查的局限**：如果 ESP32 和 APP 从同一个 IP 访问（都在本地测试），多个设备会分不清。生产环境每个 ESP32 独立 IP，没问题。
- **challenge-response 未启用**：`/api/v1/device/challenge` 和 `/verify` 已实现但当前硬件不走这条路径。
