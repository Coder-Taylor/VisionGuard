# 面向视障与老年群体的胸挂式智能设备需求规格说明书

## 总况

### 一、总体业务边界原则

- **硬件主导、本地优先**：所有安全告警必须离线可用，不依赖云端与 APP。
- **软硬严格分离**：硬件只负责感知、计算、播报，聚焦核心安全与交互功能；软件只负责配置、展示、监护，不参与本地安全判断。
- **云端只做中转与增强**：不参与硬件本地判断，不控制硬件行为，仅承担数据中转、存储及智能增强服务，降低核心功能依赖风险。
- **数据隔离**：一设备绑定一账号，数据专属转发、互不串扰。
- **全 HTTP 通信**：统一协议、降低耦合、便于开发与调试。

### 二、硬件端业务边界

**硬件只负责做这些：**

- 上电、联网、上传设备 ID 及安全认证信息，完成设备合法性校验前置操作
- 传感器采集（MPU6050 / VL53L5CX雷达 / 北斗定位）
- **本地运动检测 → 自动开启避障与摔倒**
- **本地避障判断 → 本地扬声器告警**，语音清晰提示障碍物方位，助力视障群体规避风险
- **本地摔倒判断 → 本地扬声器告警**，播放紧急提示音，同时缓存告警数据，联网后自动上报
- 按键 A/语音触发（拍照识别）
- K210 视觉计算（障碍物方位检测、药品图片拍摄、图片分块传输）
- 数据封装上报（JSON告警数据 / 药品图片流 / 音频流），联网后按协议规范上传至云端
- 接收云端LLM用药建议文本 → 本地语音逐字播报
- 北斗定位采集与缓存

### 三、监护端 APP 业务边界（纯软件、纯展示）

**APP 只负责做这些：**

- 用户注册、登录、找回密码
- 录入老人信息
- 搜索在线设备、发起绑定 / 解绑
- 展示告警历史（摔倒 / 避障 / 药品识别）
- 展示定位信息、硬件数据
- 上传配置信息到云端
- 接收云端告警推送，支持弹窗、铃声双重提示，确保监护人及时知晓危险情况

### 四、云端业务边界（中转 + 增强 + 存储）

**云端只负责做这些：**

- 提供 HTTP 接口，接收硬件 / APP 数据
- 设备上下线、心跳管理，实时监控设备运行状态，为绑定关系维护提供支撑
- 建立设备 ID ↔ 账号唯一绑定
- 数据存储（告警、定位、事件、图片、音频）
- **按绑定关系双向转发数据**
- 药品图片OCR文字识别
- 结合OCR结果调用LLM，智能生成药品用法、用量、禁忌等用药建议
- 向绑定监护人APP推送摔倒、避障危险告警

## 1. 项目概述

### 1.1 项目背景

视障人群独立出行存在避障、摔倒、物品识别三大刚需，现有产品依赖云端算力、离线不可用、响应慢。本项目采用硬件端智能优先、监护人 APP 远程监护架构，所有安全功能离线可用，联网后仅做数据上报与告警推送，符合物联网大赛 "硬件为主、软件为辅" 的评分导向。

### 1.2 术语定义

| 术语 | 定义 |
| --- | --- |
| **HTTP 通信** | 硬件、监护端、云端全部使用 HTTP 协议传输数据 |
| **写死配网** | 硬件内置固定 WiFi 名称 / 密码，用于大赛调试演示 |
| **设备绑定** | 云端建立设备 ID ↔ 监护账号唯一映射，数据不串扰 |
| **离线自治** | 无网络时，硬件避障、摔倒、扬声器告警正常运行 |
| **透传 / 转发** | 云端按绑定关系，仅向对应监护端 / 硬件发送数据 |
| **安全 Token** | 硬件通过唯一码与固定密钥异或生成，用于设备注册鉴权 |

## 2. 系统架构与配网流程

本系统整体分为**硬件终端**、**云端服务器**、**监护端**三部分。

- **硬件端**：ESP32 主控 + 电源开关 + MPU6050 + VL53L5CX 雷达 + SR2631Z3 定位 + 单按键A + MAX98357A音频功放扬声器 + K210 视觉模块，聚焦本地感知、计算与告警，所有安全功能离线可用。
- **云端**：HTTP接口服务 + 数据库存储 + OCR文字识别 + LLM大模型用药建议生成，仅承担数据中转、存储与智能增强，不参与本地安全判断。
- **监护端（APP）**：设备绑定界面、老人信息配置、危险历史记录、告警消息推送，纯软件展示与配置，不干预硬件本地操作。

**设计原则：**

- **硬件绝对主导**：所有安全功能（避障 / 摔倒）本地运行，不依赖网络
- **软硬严格分离**：
  - 硬件：只做采集、计算、告警、播报、上报
  - 监护端：只做展示、配置、绑定
  - 云端：只做接收、存储、处理、转发、推送
- **全 HTTP 通信**：无其他协议，开发简单、演示稳定
- **绑定隔离**：一个设备只绑定一个账号，数据互不干扰

### 2.1 硬件组成

硬件端包含：

| 模块 | 说明 |
| --- | --- |
| **主控** | ESP32（负责WiFi联网、传感器调度、串口通信、音频播放、HTTP数据上报） |
| **电源** | 总开关（控制整机通电 / 断电） |
| **姿态传感器** | MPU6050（运动检测、摔倒检测） |
| **测距雷达** | VL53L5CX（避障检测，获取障碍物距离） |
| **定位模块** | SR2631Z3（北斗定位，获取经纬度信息，随告警事件上报） |
| **交互** | 物理按键A（药品拍照识别触发）、麦克风（环境录音采集） |
| **音频** | MAX98357A功放+扬声器，实现离线告警音、方位提示、AI用药建议播报 |
| **视觉** | K210+摄像头，本地障碍物方位识别、药品拍摄、图片分块串口传输 |

### 2.2 配网与绑定流程

1. **开机**：打开电源开关，设备通电，自动启动所有本地核心功能（运动检测、避障检测、摔倒检测），确保离线安全可用
2. **默认配网**：ESP32 优先尝试连接内置写死的默认热点（用于大赛调试演示），连接成功后自动进入联网状态；若连接失败，不影响本地安全功能运行，仅无法上传数据与接收用药建议
3. **设备注册**：ESP32 连网后，主动上传 deviceId、uniqueCode、token（异或加密）至云端，云端校验设备合法性，校验通过后完成设备注册，标记为在线状态
4. **监护端绑定**：监护端从云端拉取在线设备列表，用户在设置界面选取设备绑定/解绑
5. **云端映射**：建立"Device_ID"与"监护端账号"的一对一映射关系，绑定成功后设备进入在线工作模式

## 3. 功能模块详细设计

### 3.1 功能模块列表

- **P0（必选 / 离线可用）**：通电启动、运动检测、本地避障、本地摔倒、本地扬声器告警，核心安全功能，确保视障与老年群体离线状态下的安全保障。
- **P1（联网可选）**：设备注册绑定、数据云端上报、APP 远程告警、按键 A 药品拍照、K210 视觉传输、药品 OCR 识别、LLM 用药建议生成、文本语音播报、环境录音上传，联网后增强功能，提升用户体验与监护便利性。

### 3.2 P0 级：自治功能

#### 一、监护端（APP）P0 级功能

##### 101 注册

- **功能描述**：监护人通过账号密码完成注册，密码满足复杂度要求，并填写被监护人基本信息。
- **触发条件**：用户打开 APP，进入注册界面。
- **处理机制**：用户输入账号、密码、确认密码、被监护人信息 → 系统校验账号唯一性、密码一致性 → 验证通过则创建账号并保存信息，同步上传至云端备案；验证失败则提示具体错误（如账号已存在、密码不一致）。

##### 102 登录

- **功能描述**：已注册用户使用账号 + 密码登录，支持单设备长期保持登录状态。
- **触发条件**：用户输入账号与密码，点击登录。
- **处理机制**：系统校验账号密码合法性 → 验证通过则登录成功，本地记录登录状态（长期有效），同步拉取绑定设备信息与历史告警数据；验证失败则提示"账号或密码错误"。

##### 103 找回密码

- **功能描述**：已注册用户可通过验证账号和原密码完成密码重置操作。
- **触发条件**：用户在登录页点击 "忘记密码 / 找回密码"。
- **处理机制**：输入注册账号 → 输入原密码 → 系统校验账号与原密码一致性 → 验证通过后允许设置新密码并确认 → 密码更新成功，同步更新云端密码信息；验证失败则提示"账号或原密码错误"。

##### 104 信息配置

- **功能描述**：在监护端录入老人基本信息，用于后续监护与告警。
- **触发条件**：用户进入 APP 设置 / 信息配置界面，点击"编辑"按钮。
- **处理机制**：用户填写信息并确认 → 本地保存配置信息，同时上传至云端备案；后续可随时修改，修改后自动同步至云端。

##### 105 告警查看

- **功能描述**：展示摔倒、避障、药品识别历史记录，包含定位信息、硬件传感数据、事件发生时间，便于监护人追溯事件详情。
- **触发条件**：用户进入告警 / 历史记录界面。
- **处理机制**：监护端向云端请求当前账号绑定设备的历史数据 → 云端按时间排序返回数据（最新事件优先） → 监护端以列表形式展示，点击单条记录可查看详情（含告警类型、发生时间、定位、硬件数据）。

##### 106 硬件绑定界面

- **功能描述**：搜索在线设备、执行绑定 / 解绑操作，显示设备在线 / 离线状态与绑定状态。
- **触发条件**：用户进入设备绑定管理界面。
- **处理机制**：监护端向云端请求在线设备列表 → 云端返回当前在线的设备信息（设备 ID、设备状态） → 监护端展示设备列表，用户选择目标设备发起绑定 / 解绑请求 → 云端校验设备状态并执行操作，操作完成后同步更新监护端设备状态（已绑定 / 未绑定、在线 / 离线）。

#### 二、硬件端 P0 级功能

##### M01 电源与配网

- **功能描述**：硬件开关控制整机上电；采用内置默认 WiFi 自动联网；上电后自动向云端上传 Device_ID 及安全认证信息，完成设备注册前置操作；若联网失败，不影响本地核心安全功能运行。
- **触发条件**：用户打开硬件电源开关。
- **处理机制**：整机上电启动 → ESP32 自动初始化各传感器与模块 → 自动连接内置固化 WiFi → 联网成功后通过 HTTP 上传设备唯一 ID、uniqueCode、token 至云端，完成合法性校验前置；联网失败则提示"联网失败，核心功能正常运行"，继续执行本地安全检测。

##### M02 本地运动感知

- **功能描述**：MPU6050 实时检测人体运动状态，运动时自动开启避障与摔倒检测模式，静止时进入低功耗模式，节省电量，确保设备续航能力。
- **触发条件**：硬件上电后持续运行。
- **处理机制**：MPU6050 实时采集三轴加速度与姿态数据 → 本地芯片（ESP32）解析数据，判断人体是否处于运动状态 → 运动状态则启动避障检测与摔倒检测，传感器保持高频率采集；静止状态则降低传感器采集频率，进入低功耗模式，延长设备续航。

##### M03 本地避障告警

- **功能描述**：VL53L5CX 激光雷达实时检测前方障碍物距离，当距离低于预设安全阈值时，硬件本地直接通过扬声器播放告警提示音，同时语音提示障碍物方位（如"左前方有障碍物"），助力视障群体规避风险。
- **触发条件**：雷达测得障碍物距离 ＜ 预设阈值。
- **处理机制**：VL53L5CX 雷达实时测距，数据实时传输至 ESP32 → ESP32 本地判断距离是否低于阈值 → 低于阈值则立即驱动 MAX98357A 功放与扬声器，播放避障告警音及方位提示，离线状态可正常运行，无需依赖网络。

##### M04 摔倒检测模块

- **功能描述**：MPU6050 依据姿态角度超限 + 合加速度突变双重条件判定摔倒，判定成功后，硬件本地立即通过扬声器播放紧急告警音（如"摔倒危险"），同时缓存摔倒数据，联网后自动上报云端，确保用户安全。
- **触发条件**：姿态异常 + 合加速度突变同时满足。
- **处理机制**：MPU6050 实时采集姿态与加速度数据，ESP32 本地实时解算 → 当姿态角度超限且合加速度出现突变时，判定为摔倒 → 立即驱动扬声器播放紧急告警音，同时缓存摔倒时间、姿态数据、定位数据 → 联网后自动封装数据上报云端，离线状态下告警功能正常运行。

##### M05 北斗定位采集

- **功能描述**：SR2631Z3 北斗定位模块实时获取经纬度信息，本地缓存定位数据（每30秒更新一次），联网后随告警事件（摔倒、避障、药品识别）同步上报云端，为监护人提供用户位置参考。
- **触发条件**：硬件上电后持续运行，无需人工触发。
- **处理机制**：定位模块自动搜星，获取经纬度信息 → 实时传输至 ESP32，ESP32 本地缓存定位数据 → 当发生摔倒、避障告警或药品识别事件时，若设备联网，同步将定位数据与事件数据一起上报云端；若离线，定位数据持续缓存，联网后自动补传。

### 3.3 P1 级：云端增强视觉功能

#### 一、云端 P1 功能

##### 201 HTTP 接口服务

- **功能描述**：提供全系统统一 HTTP 数据接口，负责接收硬件、监护端的所有数据请求与响应，统一协议规范，降低耦合，便于开发、调试与后期维护。
- **触发条件**：硬件上报数据（告警、定位、图片、音频）、监护端请求数据（历史告警、设备列表）、绑定 / 解绑、配置操作等。
- **处理机制**：云端提供标准化 HTTP 接口，接收硬件 / APP 发送的 HTTP 请求 → 解析请求参数，校验数据合法性 → 将请求分发至对应业务模块（如设备管理、数据存储、OCR 识别） → 业务模块处理完成后，生成响应数据（JSON 格式） → 通过 HTTP 协议返回至请求方（硬件 / APP）。

##### 202 设备上下线管理

- **功能描述**：维护设备唯一 ID、在线 / 离线状态、心跳记录，实时更新设备可用性，为数据转发、告警推送提供支撑，确保监护人及时知晓设备运行状态。
- **触发条件**：硬件上报心跳（每30秒一次）、硬件长时间无心跳（超过5分钟）、硬件重新联网。
- **处理机制**：硬件联网后，每30秒向云端上报一次心跳数据，告知设备在线状态 → 云端接收心跳数据，更新设备在线状态与心跳时间戳；若超过5分钟未收到设备心跳，云端标记设备为离线状态，同步推送"设备离线"提示至监护端 APP；设备重新联网后，上报心跳数据，云端更新设备状态为在线，同步推送"设备在线"提示至监护端 APP。

##### 203 设备绑定管理

- **功能描述**：建立设备 ID 与监护人账号一对一映射关系，确保数据隔离、互不串扰。
- **触发条件**：监护端发起绑定 / 解绑操作。
- **处理机制**：监护端发起绑定请求，携带账号信息与目标设备 ID → 云端校验设备是否在线、是否已绑定其他账号 → 若设备在线且未绑定，建立设备 ID 与账号的唯一映射关系，存储绑定记录，返回绑定成功响应；若设备已绑定其他账号，返回"设备已绑定其他账号"提示；监护端发起解绑请求，云端解除设备 ID 与账号的映射关系，删除绑定记录，返回解绑成功响应，设备停止向原账号推送数据。

##### 204 数据存储

- **功能描述**：存储告警数据、定位信息、用户配置、药品识别日志、图片、音频等全量系统数据，长期留存（默认留存1年），支持历史查询与追溯，为监护人提供事件回顾依据。
- **触发条件**：硬件上报事件（摔倒、避障、药品识别）、定位数据、图片、音频；监护端上传配置信息、用户信息。
- **处理机制**：云端接收来自硬件端和监护端的上报数据 → 按数据类型分类存储至 PostgreSQL 数据库 → 建立数据索引（按设备 ID、时间、事件类型） → 提供查询接口供监护端按条件检索历史数据。

##### 205 数据双向转发

- **功能描述**：按照设备 ID 与账号的绑定关系，将数据精准转发至对应 APP 或硬件，多设备间不串数据。
- **触发条件**：硬件上报告警 / 视觉数据 / 音频数据；云端下发指令 / 用药建议文本；监护端上传配置信息。
- **处理机制**：云端接收数据后，根据数据中的设备 ID 或账号信息，查询绑定关系 → 确定数据接收方（对应监护端 APP 或硬件） → 按 HTTP 协议定向转发数据，确保数据仅在绑定的设备与账号间流转，不泄露至其他账号或设备。

##### 206 摔倒监护服务

- **功能描述**：当云端接收到硬件上报的摔倒、避障危险告警数据时，立即向对应绑定的监护人 APP 推送实时告警信息，确保监护人及时知晓危险情况，快速响应。
- **触发条件**：云端收到 fall（摔倒）或 evasion（避障危险）类型上报数据。
- **处理机制**：云端解析上报的告警数据，提取设备 ID、告警类型、定位信息、硬件数据 → 根据设备 ID 查询绑定的监护人账号 → 生成告警推送内容（含老人信息、告警类型、定位、硬件数据） → 通过 HTTP 协议向监护人 APP 推送告警通知，同时触发 APP 弹窗与铃声提示，确保监护人及时看到。

##### 207 药品 OCR 文字识别

- **功能描述**：对硬件上传的药品图片进行 OCR 文字识别，提取药品说明书中的关键信息（药名、规格、用法、用量、禁忌等），为 LLM 用药建议生成提供可靠基础。
- **触发条件**：硬件完成药品图片 HTTP 上传，云端接收到图片流数据。
- **处理机制**：云端接收硬件上传的 JPEG 药品图片流 → 对图片进行预处理（去噪、裁剪，聚焦药品说明书区域） → 调用 OCR 文字识别算法，提取图片中的文字信息 → 对识别结果进行筛选、整理，结构化输出药名、规格、用法、用量、禁忌等关键信息 → 传递至 LLM 模块，用于生成用药建议。

##### 208 LLM 用药建议生成

- **功能描述**：以 OCR 识别到的药品文字为基础，结合医疗常识与老人信息（病史、过敏史），调用 LLM 大模型，智能生成通俗易懂、适合老年人阅读收听的用药建议，包含用法、用量、禁忌、注意事项等核心内容，语言简洁、咬字清晰，适配老年群体认知水平。
- **触发条件**：OCR 文字识别完成，云端获取结构化的药品文字信息。
- **处理机制**：云端接收 OCR 结构化文本，同时获取该设备绑定的老人信息（病史、过敏史） → 调用 LLM 大模型，输入 OCR 文本与老人信息，设置"老年友好、简洁易懂"的生成规则 → 大模型智能生成用药建议文本（不含专业术语，语速适配老年群体） → 云端缓存用药建议文本，同时提供 HTTP 接口，供硬件拉取文本用于本地播报。

#### 二、硬件端 P1 功能

##### M06 药品识别模块

- **功能描述**：按键 A 为药品识别专属触发键，同时支持语音指令触发（如"识别药品"），操作简洁，适配视障与老年群体；ESP32 下发拍照指令至 K210；K210 完成药品图片拍摄、压缩分块传输；ESP32 接收完整图片后上传至云端；云端完成 OCR 识别与 LLM 用药建议生成；硬件从云端拉取用药建议文本，本地语音逐字播报，助力老人正确用药。
- **触发条件**：用户按下按键 A 或说出语音指令"识别药品"。
- **处理机制**：
  1. ESP32检测按键A按下或语音指令触发，串口发送指令 `$TAKE_PHOTO!`
  2. K210接收指令，拍摄药品图片并压缩为JPEG格式
  3. K210分块传输图片数据至ESP32
  4. ESP32拼接完整图片，通过HTTP上传至云端
  5. 云端执行OCR药品文字识别 + LLM用药建议生成
  6. 硬件请求获取结果文本，驱动扬声器逐字播报用药指导

##### M07 定位上传

- **功能描述**：硬件端实时获取北斗定位信息，联网状态下，定时（每5分钟）或事件触发（摔倒、避障、药品识别）时上传至云端，与告警事件进行绑定关联，便于监护人查看事件发生位置。
- **触发条件**：设备已绑定、设备联网。
- **处理机制**：硬件通过 SR2631Z3 模块获取当前经纬度 → 定时（每5分钟）或事件触发时，由 ESP32 封装定位数据 → 通过 HTTP 协议上传至云端 → 云端将定位数据与对应设备、事件进行绑定存储，同步推送至监护端 APP 展示。

##### M08 告警数据上报

- **功能描述**：摔倒、避障事件封装为 JSON 格式，携带定位信息上传云端。
- **触发条件**：本地判定摔倒或避障事件后，设备联网状态下自动触发。
- **处理机制**：ESP32 将事件类型、设备ID、传感器数据、北斗定位封装为标准 JSON → 通过 HTTP POST 上传至云端 `/upload_alarm` 接口。

##### M09 本地语音播报

- **功能描述**：离线告警音（避障、摔倒）、障碍物方位提示音本地独立播放，无需依赖网络；联网场景下，接收云端下发的 LLM 用药建议长文本，通过本地汉字 WAV 语音库逐字拆分、拼接播放。
- **触发条件**：本地避障 / 摔倒判定成功（离线播报）、硬件拉取到云端用药建议文本（联网播报）。
- **处理机制**：离线场景下，ESP32 判定避障或摔倒后，直接调用本地预存的告警音、方位提示音，通过 MAX98357A 功放驱动扬声器播放；联网场景下，ESP32 接收云端用药建议文本后，逐字拆分文本，匹配本地汉字 WAV 语音库，拼接语音片段，驱动扬声器逐字播报，确保播报清晰、易懂。

##### M10 环境录音上传

- **功能描述**：麦克风检测环境音量超标，静音延时到达后自动录制并上传音频数据至云端存档。
- **触发条件**：环境音量超过阈值且持续一段时间后自动触发。
- **处理机制**：麦克风实时检测环境音量 → 音量超标则开始录音（WAV格式） → 静音延时到达后停止录音 → ESP32 通过 HTTP 上传音频二进制流至云端 `/upload_audio` 接口。

## 4. 数据通信与监护协议

### 4.1 监护端（App）功能

#### 4.1.1 界面功能

**主界面：**

负责显示摔倒或避障的危险提醒及历史信息，包含摔倒 / 避障危险的硬件信息（设备 ID、告警类型）、硬件定位信息（经纬度）、事件发生时间，界面简洁、字体较大，适配老年监护人查看。

**设置界面：**

负责设置老人的基本信息（姓名、年龄、病史、过敏史等）以及硬件的绑定 / 解绑，设置完成后可将老人基本信息以及硬件配置信息发送到云端，同步更新备案。

### 4.2 云端（后端）逻辑

#### 4.2.1 数据路由

**接收数据类型：**

- ESP32 上报 JSON 数据（摔倒告警、避障告警、设备注册、定位数据）、JPEG 药品图片二进制流、WAV 音频流
- APP 上报用户信息（注册、登录、老人信息）、绑定 / 解绑请求、配置信息

**处理逻辑：**

- 对上报数据进行解析与校验
- 摔倒 / 避障事件 → 触发 APP 告警推送
- 建立设备 ID 与监护人账号的一对一映射关系
- 确保多设备环境下数据严格隔离、互不串扰
- 根据绑定关系将硬件上报的数据转发至对应监护端 APP，将 APP 上传的配置信息转发至对应硬件

#### 4.2.2 增强服务

- **药品OCR文字识别**：解析硬件上传的药品图片，提取说明书、药名、规格、用法、用量、禁忌等关键文字，确保识别精准，为用药建议生成提供可靠基础。
- **LLM智能用药建议**：基于 OCR 识别结果，结合老人病史、过敏史，调用 LLM 大模型，生成老年友好型用药提示文本，语言简洁、通俗易懂，避免专业术语，适配老年群体认知水平。
- **数据存储**：永久存储设备告警、定位、事件、药品识别图片与 AI 建议记录（默认留存1年），支持历史查询与追溯，为监护人提供事件回顾依据。

### 4.3 摔倒监护服务（APP 告警服务）

#### 4.3.1 触发条件

云端接收到硬件上报的 fall（摔倒）或 evasion（避障危险）数据包。

#### 4.3.2 推送对象

已与设备完成绑定的监护人 APP 账号。

#### 4.3.3 推送内容

- 老人基本信息：姓名、年龄、病史（简要）等
- 危险类型：摔倒 / 靠近障碍物
- 硬件数据详情：MPU6050 数据 / 雷达距离数据
- 定位信息：硬件北斗定位（经纬度）

## 5. 数据接口与通讯协议规范

> 本章节定义硬件端（ESP32/K210）向云端上报数据的标准 JSON 格式、HTTP 接口地址、请求方式、串口通讯协议，所有硬件开发与后端对接严格遵循此字段与交互定义。
>
> 详细接口文档参见 **api.md**。

### 5.1 摔倒检测上报（M04）

- **触发条件**：MPU6050 检测设备姿态倾角超出设定安全阈值，判定为摔倒事件，本地语音告警并上传云端。
- **接口地址**：`http://172.20.10.3:8888/upload_alarm`
- **请求方式**：POST
- **请求头**：`Content-Type: application/json`

**JSON报文示例：**

```json
{
  "deviceId": "DEVICE_001",
  "alarmType": "fall",
  "angleX": -48.25,
  "angleY": 12.36,
  "longitude": 115.236589,
  "latitude": 28.369852
}
```

**字段说明：**

| 字段名 | 数据类型 | 字段说明 |
| --- | --- | --- |
| deviceId | 字符串 | 硬件固定唯一设备标识 |
| alarmType | 字符串 | 事件类型，摔倒固定为 fall |
| angleX | 浮点型 | X轴倾斜角度，保留2位小数 |
| angleY | 浮点型 | Y轴倾斜角度，保留2位小数 |
| longitude | 浮点型 | 北斗模块实时经度，保留6位小数 |
| latitude | 浮点型 | 北斗模块实时纬度，保留6位小数 |

### 5.2 避障检测上报（M03）

- **触发条件**：VL53L5CX激光雷达检测前方障碍物距离小于安全阈值，触发避障告警，同步上传云端。
- **接口地址**：`http://172.20.10.3:8888/upload_alarm`
- **请求方式**：POST
- **请求头**：`Content-Type: application/json`

**JSON报文示例：**

```json
{
  "deviceId": "DEVICE_001",
  "alarmType": "obstacle",
  "lidarDist": 456,
  "longitude": 115.236589,
  "latitude": 28.369852
}
```

**字段说明：**

| 字段名 | 数据类型 | 字段说明 |
| --- | --- | --- |
| deviceId | 字符串 | 硬件固定唯一设备标识 |
| alarmType | 字符串 | 事件类型，避障固定为 obstacle |
| lidarDist | 整型 | 雷达检测最小障碍物距离，单位 mm |
| longitude | 浮点型 | 北斗模块实时经度，保留6位小数 |
| latitude | 浮点型 | 北斗模块实时纬度，保留6位小数 |

### 5.3 图像上传接口（拍照/OCR识别）

- **触发条件**：按下按键A，ESP32向K210下发拍照指令，接收分块图片数据完成后自动上传。
- **接口地址**：`http://172.20.10.3:8888/upload_image`
- **请求方式**：POST
- **请求头**：`Content-Type: image/jpeg`
- **传输形式**：无JSON报文，直接传输 JPEG 图片二进制裸流。

### 5.4 OCR 文本获取接口

- **触发条件**：图片上传完成后，硬件主动拉取云端OCR识别结果。
- **接口地址**：`http://172.20.10.3:8888/get_ocr_text`
- **请求方式**：GET
- **返回内容**：纯文本字符串
- **业务逻辑**：硬件获取文字后，调用本地WAV音频库逐字语音播报。

### 5.5 环境音频上传接口

- **触发条件**：麦克风检测环境音量超标，静音延时到达后自动录制并上传音频数据。
- **接口地址**：`http://172.20.10.3:8888/upload_audio`
- **请求方式**：POST
- **请求头**：`Content-Type: application/octet-stream`
- **传输形式**：直接传输 WAV 音频二进制流。

### 5.6 设备注册认证接口

- **接口地址**：`http://172.20.10.3:8888/device_register`
- **请求方式**：POST
- **请求头**：`Content-Type: application/json`

**JSON报文示例：**

```json
{
  "deviceId": "DEVICE_001",
  "uniqueCode": "DEVICE_2026_ESP32_K210",
  "token": "加密异或校验字符串"
}
```

### 5.7 设备绑定查询接口

- **接口地址**：`http://172.20.10.3:8888/check_bind`
- **请求方式**：GET
- **请求参数**：`?deviceId=DEVICE_001`
- **判定规则**：响应包含 `bind_success` 即为绑定成功，设备进入在线工作状态。

### 5.8 ESP32 ↔ K210 串口通讯协议

1. **串口参数**：波特率115200、8N1；ESP32 UART2，RX=3、TX=4
2. **ESP32下发指令**：
   - `$TAKE_PHOTO!` ：触发K210拍照与图像传输
3. **K210上行指令**：
   - 方位指令：`$LEFT!` / `$MID!` / `$RIGHT!`
   - 图片传输：`$IMG_START:长度!` + 二进制分块数据 + `$IMG_END!`

### 6.1 药品识别业务流程

硬件按键触发拍照 → K210采集图像 → ESP32上传图片二进制流 → 云端OCR提取文字 → 硬件拉取识别文本 → 本地音频模块语音朗读播报。

### 6.2 数据库设计建议

系统采用 **PostgreSQL** 关系型数据库进行结构化数据存储，统一管理设备绑定、事件记录、用户信息，保证数据一致性、事务完整性与关联查询能力。

| 字段名 | 类型 | 说明 |
| --- | --- | --- |
| user_id | VARCHAR(64) | 用户唯一ID (账号) |

| phone | VARCHAR(16) | 紧急联系人手机号 |
| elder_info | TEXT | 老人基本信息 |

| 字段名 | 类型 | 说明 |
| --- | --- | --- |

### 6.3 摔倒 / 避障告警推送逻辑

**触发条件**：云端接口接收到 `/upload_alarm` 推送的 fall、obstacle 类告警数据。

**处理逻辑**：

1. 通过 device_id 关联查询 PostgreSQL 用户绑定表
2. 匹配对应老人信息、紧急联系人、实时定位与传感数据
3. 向绑定监护人APP推送告警消息

**推送内容**：

- 老人基本信息
- 危险类型：摔倒告警 / 障碍物靠近告警
- 硬件传感实时数据
- 北斗高精度经纬度
- 提示文案：请及时查看当事人情况

## 7. 系统运行流程

### 7.1 正常联网运行流程

1. 设备上电自动连接WiFi，完成云端设备注册与绑定状态校验
2. 后台常驻运行MPU6050姿态检测、VL53L5CX雷达避障、北斗定位解析
3. 检测到摔倒、障碍物等危险事件，本地离线语音即时提醒，同时POST上传告警数据至云端
4. 监护人APP接收实时告警推送，查看定位与设备数据
5. 按下按键A，ESP32发送串口指令控制K210拍照，图片分块接收完毕后上传云端
6. 云端完成OCR文字解析，硬件拉取文本内容，通过音频模块完成语音播报

### 7.2 离线运行流程

- 断网状态下，摔倒检测、雷达避障、障碍物方位语音提示、姿态监测全部离线独立运行
- 禁止所有HTTP云端上传、数据入库、APP消息推送等联网功能
- 图片上传、音频上传、云端OCR文字识别等依赖网络的功能暂停使用
- 本地安全防护核心功能不受网络中断影响，保障基础监护能力

## 8. 非功能性需求

### 8.1 性能需求

- 本地响应：本地避障 / 摔倒检测响应时间 ≤ 0.5 秒
- 网络延迟：硬件到云端数据传输延迟 ≤ 2 秒
- 系统可用性：核心安全功能可用性 ≥ 99.9%

### 8.2 安全需求

- 数据隔离：不同设备间数据完全隔离，互不串扰
- 隐私保护：老人个人信息加密存储
- 通信安全：所有网络通信采用 HTTP 安全传输

### 8.3 可靠性需求

- 本地优先：核心安全功能离线可独立运行
- 数据完整：告警数据不丢失，支持网络恢复后重传
- 容错机制：网络异常时硬件缓存数据，恢复后自动续传
