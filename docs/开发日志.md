## 2026-05-06 深夜 — OCR GET 401 修复 + drugName UTF-8 截断 + 云端部署迁移

### OCR 硬件轮询 GET 401 根因发现与修复
- **现象**：设备 JWT 可成功 POST `/api/v1/device/ocr/image` 上传图片，但 GET `/api/v1/ocr/result/latest?deviceId=xxx` 返回 401。Android APP 显示"等待中"。
- **根因**：Fiber 路由匹配顺序 bug。`/api/v1/ocr/result/:taskId`（参数路由, userAuth）注册在 `/api/v1/ocr/result/latest`（精确路由, deviceAuth）之前，Fiber 将 `latest` 当作 `:taskId` 值，路由到 userAuth 中间件 → 设备 JWT 通不过用户认证。
- **修复**：精确路由 `latest` 移到参数路由 `:taskId` 之前（`backend/cmd/server/main.go` + `cloud-deploy/cmd/server/main.go`）

### drugName UTF-8 截断修复
- **现象**：豆包返回中文 drugName（如"阿莫西林胶囊"），byte-based `drugName[:20]` 在中间截断多字节 UTF-8 字符 → PostgreSQL 报错 `invalid byte sequence for encoding "UTF8"`
- **修复**：改为 rune-based `[]rune(drugName)[:20]`（`backend/internal/service/medication.go` + `cloud-deploy` 同步）

### 云端部署迁移
- 旧容器名 `visionguard-*` → 新 `cloud-deploy-*`，数据卷 `pgdata` → `visionguard_pgdata`
- `.env` 从旧容器环境变量提取（`docker inspect visionguard-backend-1`）
- 部署命令：`docker compose -f docker-compose.prod.yml up -d --build`（注意空格，非连字符）
- 旧容器端口冲突需先 stop/rm

## 2026-05-05（深夜） — 真机网络连通修复 + 三端联调就绪

### 问题发现

Android APP 在手机上操作时，创建老人、修改密码、换绑手机号全部失败（提示"创建失败"）。此前 `test_e2e.sh` curl 模拟全部通过，误以为全链路正常。

### 根因定位

```
手机开热点(Honor400) → 电脑连热点(10.26.43.176) → 手机 APP 访问后端
                                                     ↑
                              Android 热点模式下手机自身流量走蜂窝网(rmnet_data0)
                              不走热点接口(wlan2)，ping 10.26.43.176 100% 丢包
```

诊断命令：
```bash
adb shell "ip addr show" | grep "inet "   # 发现 wlan2=10.26.43.244, rmnet_data0=10.152.87.91
adb shell "ping -c 2 10.26.43.176"        # 100% 丢包，确认网络不通
```

这是 Android 系统限制，非代码 bug。此前 e2e 测试全在电脑本地 curl 完成，从未经过手机热点路径。

### 解决方案

ADB 反向隧道：
```bash
adb reverse tcp:3000 tcp:3000   # 手机 127.0.0.1:3000 → 电脑 localhost:3000
```

Android `RetrofitClient.kt` BASE_URL 改为 `http://127.0.0.1:3000/`。

安装 ADB：下载 `platform-tools-latest-windows.zip` → 解压到 `C:\Users\HONOR\AppData\Local\Android\Sdk\platform-tools\`。

构建 debug APK 以启用 HTTP 日志，增加详细错误诊断信息（`errorBody()` + HTTP 状态码）。

### 晚上：硬件接口验证 + 修复

对比 ESP32 固件源码（`hardware/sketch_apr23a.ino`）逐接口验证，发现 3 处后端实现与硬件行为不匹配：

| 接口 | 问题 | 根因 |
|------|------|------|
| `POST /upload_image` | 后端用 `c.Query("deviceId")` 取参 | 硬件不传 deviceId（api.md 规定"请求参数: 无"），只发 JPEG 二进制流 |
| `GET /get_ocr_text` | 后端用 `c.Query("deviceId")` 取参 | 硬件不传任何参数（api.md 规定"请求参数: 无"） |
| `POST /upload_audio` | 后端用 `c.Query("deviceId")` 取参 | 硬件不传 deviceId，只发 WAV 二进制流 |

**修复方案**：IP 反查设备
- 设备首次调用 `/device_register` 时，后端记录客户端 IP 到 devices 表
- 三个无 deviceId 的接口通过 `GetDeviceIDByIP(c.IP())` 反查设备 ID
- 查不到时降级：图片/音频存到 `"unknown"` 目录，OCR 返回全局最新记录

**修复文件**：
- `service/device.go` — `RegisterByXOR` 增加 IP 参数，新增 `GetDeviceIDByIP` 函数
- `handler/hardware.go` — 三个 handler 改为 IP 反查 deviceId
